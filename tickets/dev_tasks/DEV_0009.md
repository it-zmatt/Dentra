# DEV_0009 — Implement Rust user management commands and service wrappers

**Parent feature:** [FEAT_0003](../features/FEAT_0003.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description
Implement backend commands and frontend service wrappers for admin user management: list users, create user, update user, delete user (with safety checks), and reset password. This creates the API surface the Settings Users panel will consume.

## Scope
- **Files likely touched:** `src-tauri/src/commands/users.rs`, `src-tauri/src/main.rs`, `src/services/db.ts`, `src/types/index.ts`.
- **Boundary:** No panel UI here (DEV_0010); no sidebar/routing enforcement changes here (DEV_0011).

## Requirements / Acceptance criteria
- [ ] Add/complete Tauri commands: `get_users`, `create_user`, `update_user`, `delete_user`, `reset_user_password`.
- [ ] Enforce admin-only access for mutating commands (server-side guard, not UI-only).
- [ ] Hash passwords before storage; never store plaintext password.
- [ ] Reject duplicate email with user-facing error message.
- [ ] Prevent deleting the currently logged-in admin and prevent deleting the final admin account.
- [ ] Include `permissions` payload in create/update/get contracts.
- [ ] Register all new commands in `main.rs` and expose wrappers in `src/services/db.ts`.
- [ ] Frontend code paths continue to call service wrappers (no direct `invoke()` from components).
- [ ] Respects `.claude/CLAUDE.md` conventions (`Result<T,String>`, no `unwrap()`).

## Testing Plan

### Input
- Logged-in admin executes create/edit/delete/reset actions from invoked commands
- Logged-in non-admin attempts mutating commands

### Output
- Admin operations succeed with correct DB updates
- Non-admin operations are rejected with authorization error
- Email uniqueness and safety constraints enforced

### Test cases
1. Create user with unique email succeeds and appears in `get_users`.
2. Create user with duplicate email fails with clear error.
3. Reset password allows immediate login with new password and blocks old one.
4. Non-admin mutation attempt is rejected.
5. Attempt to delete final admin fails safely.
