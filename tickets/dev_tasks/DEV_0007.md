# DEV_0007 — Wire license gate and startup routing in App.tsx

**Parent feature:** [FEAT_0002](../FEAT_0002.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description

`App.tsx` currently loads all data stores and renders the page stubs with an empty sidebar. This DEV wires up the full startup sequence: check license validity → attempt auto-login → show the appropriate screen (license gate, login screen, or app shell). It also creates an inline `LicenseGatePage` component (small enough to live in `App.tsx` or a sibling file) that shows the machine ID and a key entry field when the grace period has expired and no valid key is stored. This is the final DEV that makes FEAT_0002 testable end-to-end.

## Scope

- **Files to touch:** `src/App.tsx` — add startup sequence, conditional rendering, replace `<nav>` stub call (nav replaced by `NavSidebar` from DEV_0006)
- **Files to create:** `src/pages/LicenseGatePage.tsx` — license entry screen (machine ID display + key field + submit)
- **Files to touch:** `src/i18n/ar.ts`, `src/i18n/fr.ts`, `src/i18n/en.ts` — add i18n keys for license gate strings
- **Boundary:** This DEV does NOT build the login form or nav sidebar — those are DEV_0005 and DEV_0006 (this DEV depends on both being done). It does NOT build the license validation Rust command — that already exists in `license.rs`. It does NOT implement user management UI.

## Requirements / Acceptance criteria

- [ ] On startup, `App.tsx` runs this sequence synchronously before rendering any page:
  1. Load `local_settings` and `global_settings`
  2. Check license: if `license_valid = 'false'` and `install_date` is more than 7 days ago, set app state to `'license-gate'`
  3. Otherwise attempt `autoLogin()` from the auth store; if it returns `false`, set app state to `'login'`
  4. If auto-login returns `true`, load all data stores in parallel and set app state to `'app'`
- [ ] While the startup sequence is running, a full-screen loading spinner is shown (not a blank screen)
- [ ] When app state is `'license-gate'`, render `<LicenseGatePage />`; no other screen is accessible
- [ ] When app state is `'login'`, render `<LoginPage />` (DEV_0005); auth store login success transitions app state to `'app'`
- [ ] When app state is `'app'`, render the app shell: `<NavSidebar />` (DEV_0006) + the current page component
- [ ] `LicenseGatePage` displays the machine ID (calls `getMachineId()` from `db.ts`) and a text field for key entry; submitting calls `validateLicenseKey(key)` from `db.ts`; on success, writes `license_valid = 'true'` and `license_key = key` to `local_settings` via `saveLocalSettings`, then transitions to `'login'`; on failure, shows inline error "Clé de licence invalide"
- [ ] The 7-day grace period is calculated as: `Date.now() - Number(install_date) > 7 * 24 * 60 * 60 * 1000`; if `install_date` is empty (first launch), write the current timestamp to `local_settings` as `install_date` and treat the license as valid for this session
- [ ] Transitioning from `'login'` to `'app'` triggers the parallel data load (`loadPatients`, `loadAppointments`, `loadDoctors`, `loadLabworks`, `loadExpenses`) — data load only happens after confirmed login, not on every startup
- [ ] All license gate UI strings use i18n keys — no hardcoded strings

## Testing Plan

### Input
- Fresh install (no `install_date` stored): app should start normally, write `install_date`, proceed to login
- Install date 3 days ago, no valid license: grace period still active, proceed to login
- Install date 8 days ago, `license_valid = 'false'`: show license gate
- Install date 8 days ago, `license_valid = 'true'`: skip license gate, proceed to auto-login check
- Valid session stored (auto-login): skip login, go directly to app shell
- No session stored: show login page

### Output
- Fresh install: loading spinner → login page
- Grace period active: loading spinner → login page
- License expired: loading spinner → license gate screen (machine ID visible, key entry field)
- Valid license, valid session: loading spinner → app shell (patients page)
- Valid license, no session: loading spinner → login page

### Test cases
1. **First launch:** No `install_date` → written, proceed to login → after login, app shell loads, data stores populated
2. **Grace period (day 3):** License not validated, but within 7 days → no license gate, proceed to login
3. **License expired:** Day 8, `license_valid = 'false'` → license gate shown, machine ID displayed
4. **Valid key entered:** Enter correct key in license gate → `license_valid` saved → transition to login
5. **Invalid key entered:** Enter wrong key → inline error, stay on license gate
6. **Auto-login:** Valid session ID in `local_settings` + matching user in DB → app shell shown, login page skipped
7. **Stale session:** `logged_in_user_id` stored but user deleted from DB → `autoLogin()` returns false → login page shown
