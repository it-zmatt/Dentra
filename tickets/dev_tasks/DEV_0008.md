# DEV_0008 — Add users schema migration and Rust model updates

**Parent feature:** [FEAT_0003](../features/FEAT_0003.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description
Add a migration that upgrades the `users` table to support per-user permissions while keeping existing clinics safe. This DEV defines the backend data contract for user onboarding and access control, including defaults for existing rows.

## Scope
- **Files likely touched:** `src-tauri/migrations/004_users_permissions.sql` (new), `src-tauri/src/types.rs` or user model files, `src-tauri/src/db/*` migration runner touchpoints if needed.
- **Boundary:** No frontend UI in this DEV; no user CRUD commands yet (handled in DEV_0009).

## Requirements / Acceptance criteria
- [ ] Add migration to extend `users` with `permissions_json TEXT NOT NULL DEFAULT '[false,false,false,false,false,false]'`.
- [ ] Migration backfills existing users safely (no row loss, no nullable permissions).
- [ ] Keep `is_admin` behavior unchanged; admin still bypasses permission checks.
- [ ] Rust user struct includes `permissions` field mapped to `Vec<bool>` (or equivalent strong type).
- [ ] Deserialization handles malformed legacy data by falling back to 6 `false` flags.
- [ ] Update docs/comments where schema assumptions changed (`docs/DB_SCHEMA.md` alignment note if needed).
- [ ] Respects `.claude/CLAUDE.md` conventions (no `unwrap()`, Result-based errors).

## Testing Plan

### Input
- Existing DB with users created before migration
- New DB initialized from scratch

### Output
- Migration runs without errors on both existing and fresh DBs
- Existing users retain login capability and receive default permission payload
- New users inserted with explicit permissions persist correctly

### Test cases
1. Existing DB migration succeeds and all users remain queryable.
2. New DB contains users schema with `permissions_json` and default value.
3. Corrupt permission payload falls back to safe denied permissions.
