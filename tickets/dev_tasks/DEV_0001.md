# DEV_0001 — Implement zip-and-write Rust command for backup export

**Parent feature:** [FEAT_0001](../features/FEAT_0001.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description

Build the Rust backend command (`export_backup`) that receives a folder path from the frontend, zips the `data.db` file and the entire `photos/` directory tree into a single `.zip` archive, and writes it to the admin-selected destination. The zip file is named `dental_backup_YYYY-MM-DD.zip` where the date is in ISO 8601 format. The operation must be synchronous and return either success with the full path to the created file, or a clear error message.

## Scope
- **Files likely touched:** `src-tauri/src/commands/backup.rs`, `src-tauri/src/main.rs` (invoke handler registration), `Cargo.toml` (add `zip` and `chrono` crates if not present)
- **Boundary:** This DEV does NOT handle:
  - Restore from backup (separate DEV)
  - Encryption of the zip file
  - Compression settings or progress callbacks
  - UI integration (that's DEV_0002)
  - i18n strings (that's DEV_0003)

## Requirements / Acceptance criteria
- [ ] `export_backup` command accepts a `destination_folder: String` parameter and returns `Result<String, String>` (success returns full path, failure returns error message)
- [ ] Command resolves `AppData/Local/DentalApp/data.db` and `AppData/Local/DentalApp/photos/` using Tauri path APIs (never hardcoded absolute paths)
- [ ] Creates a zip archive in the destination folder named `dental_backup_YYYY-MM-DD.zip` where date is today's date in ISO 8601 format
- [ ] Zip archive includes:
  - `data.db` at the archive root (not in a subdirectory)
  - `photos/` directory with entire tree of subdirectories and image files preserved
- [ ] If `photos/` directory does not exist or is empty, the backup still succeeds and includes only `data.db`
- [ ] If destination folder is not writable, returns error message: `"Cannot write to selected folder. Check permissions."`
- [ ] If destination disk is full, returns error message: `"Insufficient disk space for backup."`
- [ ] Uses the `zip` crate to create the archive (standard Rust zip library)
- [ ] Respects `.claude/CLAUDE.md` conventions: no `unwrap()`, all errors propagated as `?` or matched, Result-based error handling
- [ ] Command is registered in `main.rs` invoke handler

## Testing Plan

### Input
- Command called with `destination_folder = "D:\\USB_Backup"` (Windows USB drive path)
- `AppData/Local/DentalApp/data.db` exists (8MB)
- `AppData/Local/DentalApp/photos/` exists with structure:
  ```
  photos/
    appt_0001/
      photo1.jpg (500KB)
      photo2.jpg (600KB)
    appt_0002/
      photo1.jpg (400KB)
  ```

### Output
- Command returns: `"D:\\USB_Backup\\dental_backup_2026-04-26.zip"`
- File exists at that path with size ~9.5MB
- Zip archive can be extracted on another machine and contains the same directory structure

### Test cases
1. **Happy path:** Destination is writable, disk has space, zip is created successfully
2. **No photos directory:** `AppData/Local/DentalApp/photos/` does not exist; zip contains only `data.db`, command succeeds
3. **Empty photos directory:** `photos/` exists but has no subdirectories; zip contains `photos/` (empty dir) and `data.db`
4. **Destination not writable:** Destination folder has read-only permissions; returns error message with guidance
5. **Disk full:** Destination disk has <100MB free (smaller than db + photos); returns error message
