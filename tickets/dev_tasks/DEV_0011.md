# DEV_0011 — Enforce per-user permissions in auth/session/nav flow

**Parent feature:** [FEAT_0003](../features/FEAT_0003.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description
Wire newly stored per-user permissions into session hydration and runtime navigation checks so each regular user only sees and enters authorized modules. This replaces dependency on a single global permissions list for user-level access decisions.

## Scope
- **Files likely touched:** `src/store/authStore.ts`, `src/hooks/useAuth.ts`, `src/components/common/NavSidebar.tsx`, route guards in `src/App.tsx`, `src/types/index.ts`.
- **Boundary:** No user CRUD UI work (DEV_0010) and no backend command creation (DEV_0009).

## Requirements / Acceptance criteria
- [ ] Auth store loads and exposes current user permission flags from backend payload.
- [ ] Sidebar filtering uses `currentUser.permissions` for non-admin users.
- [ ] Route-level guard prevents manual navigation to unauthorized modules.
- [ ] Admin bypass remains intact for all modules.
- [ ] If permissions payload is missing/invalid, fail closed to denied for non-admin users.
- [ ] Existing login/auto-login flow remains stable and backward compatible.
- [ ] Add/update unit-level checks for permission mapping correctness.
- [ ] Keep logic in store/hooks/services layers; components stay thin.

## Testing Plan

### Input
- Two regular users with different permissions
- One admin user
- Auto-login and fresh-login scenarios

### Output
- Each regular user sees only allowed modules
- Unauthorized route access redirects safely
- Admin retains full access

### Test cases
1. Restricted user cannot see Doctors/Stats in sidebar.
2. Restricted user direct URL/state navigation to blocked module is denied.
3. Admin account sees all modules regardless of permission payload.
4. Auto-login restores correct permissions after app relaunch.
