# DEV_0010 — Build admin Users panel UI with create/edit/reset flows

**Parent feature:** [FEAT_0003](../features/FEAT_0003.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description
Create the admin-facing Users panel in Settings to manage staff accounts end-to-end: list users, create user, edit profile/permissions, and reset password. UI must follow existing FluentUI patterns and be fully localized (ar/fr/en).

## Scope
- **Files likely touched:** `src/pages/SettingsPage.tsx`, `src/components/settings/*`, `src/store/usersStore.ts` (or equivalent), `src/hooks/useUsers.ts`, `src/i18n/ar.ts`, `src/i18n/fr.ts`, `src/i18n/en.ts`.
- **Boundary:** No Rust command changes here (DEV_0009); no global route enforcement changes here (DEV_0011).

## Requirements / Acceptance criteria
- [ ] Add admin-only Users section in Settings with table/list of users.
- [ ] Provide "Create user" form: name, email, password, permissions matrix (6 toggles).
- [ ] Provide "Edit user" form: name, email, permissions updates.
- [ ] Provide "Reset password" action requiring new password input and confirmation.
- [ ] Provide "Delete user" action with confirmation and surfaced backend safety errors.
- [ ] Uses Zustand store + hook pattern (`component -> hook -> service`).
- [ ] All visible strings are i18n keys in `ar/fr/en`; no hardcoded UI text.
- [ ] RTL layout validated for Arabic locale.
- [ ] Non-admin users never see this section.

## Testing Plan

### Input
- Admin user opens Settings and manages multiple users
- Non-admin user opens Settings

### Output
- Admin can fully manage users from one panel
- Non-admin cannot view or trigger any users management UI
- Form validations and backend errors are shown clearly

### Test cases
1. Admin creates user and sees row appear immediately.
2. Admin edits permissions and changes persist on refresh.
3. Password reset path works and shows success feedback.
4. Non-admin login confirms Users section is absent.
