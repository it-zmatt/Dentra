# DEV_0005 — Build LoginPage component with email/password form

**Parent feature:** [FEAT_0002](../FEAT_0002.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description

Build `src/pages/LoginPage.tsx` — a full-screen login form the user sees when no valid session exists. The form has an email field, a password field, and a submit button. On submit it calls `login()` from the auth store (DEV_0004). Errors from a failed login are shown inline below the form. The page is RTL-aware (Arabic default) and uses FluentUI components throughout. No routing logic lives here — the parent `App.tsx` (DEV_0007) decides when to show this page.

## Scope

- **Files to create:** `src/pages/LoginPage.tsx`
- **Files to touch:** `src/i18n/ar.ts`, `src/i18n/fr.ts`, `src/i18n/en.ts` — add i18n keys for all login screen strings
- **Boundary:** This DEV does NOT implement session persistence or auto-login logic — that is DEV_0004. It does NOT mount itself in App.tsx — that is DEV_0007. It does NOT handle the license gate screen — that is DEV_0007.

## Requirements / Acceptance criteria

- [ ] `LoginPage.tsx` renders a centered card on a neutral full-screen background
- [ ] Card contains: clinic name/logo area (reads `globalSettings.clinicName` if available, falls back to app name), email `Input`, password `Input` with show/hide toggle, submit `Button`
- [ ] Submitting calls `useAuth().login(email, password)` — shows a spinner inside the submit button while the async call is in progress
- [ ] On failed login, an inline error message appears below the form; the password field is cleared, the email field retains its value
- [ ] Error message text uses the i18n key `login.errorInvalidCredentials`; the Rust error string is NOT shown directly to the user
- [ ] On successful login, the component does nothing — `App.tsx` (DEV_0007) reacts to the auth store change and unmounts LoginPage
- [ ] Pressing Enter in either field submits the form (no mouse required)
- [ ] RTL layout: field labels and text align right when `locale = 'ar'`; card layout stays centred regardless of direction
- [ ] All text (labels, placeholder, button, error) uses i18n keys — no hardcoded French or Arabic strings
- [ ] Uses `@fluentui/react-components`: `Input`, `Button`, `Spinner`, `Text`, `Card` (or equivalent FluentUI primitives) — no custom CSS beyond Tailwind utilities

## Testing Plan

### Input
- Render `<LoginPage />` in isolation (no store pre-populated)
- Fill correct credentials and submit
- Fill wrong credentials and submit
- Tab through fields and press Enter to submit
- Switch locale to `ar` and observe layout

### Output
- Form renders with email field, password field, submit button
- Correct credentials: spinner shown → login store updated → page does nothing (parent handles transition)
- Wrong credentials: error message appears below form, password field cleared, email field intact
- Enter key submits from either field
- Arabic locale: text aligns right, layout mirrors correctly

### Test cases
1. **Happy path:** Valid credentials submitted → spinner → auth store `currentUser` populated → no crash
2. **Wrong password:** Error string shown inline, password cleared, form still interactive
3. **Network/Rust error:** If `login` invoke fails with unexpected error, generic error message shown (not a raw Rust error string)
4. **RTL check:** In `locale = 'ar'`, form fields and labels are right-aligned and the card looks correct
