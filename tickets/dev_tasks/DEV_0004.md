# DEV_0004 — Build auth Zustand store with login, logout, and session persistence

**Parent feature:** [FEAT_0002](../FEAT_0002.md)
**Status:** Open  *(Open → Dev → Dev Review → Testing → Done)*

## Description

Create `src/store/authStore.ts` — the single source of truth for who is logged in. The store exposes `login(email, password)` which calls the existing `login` invoke wrapper in `db.ts`, `logout()` which clears state and removes the persisted user ID, and `autoLogin()` which reads the stored user ID from `local_settings` and re-fetches that user from the `users` list so the app can skip the login screen on relaunch. The store does NOT manage license state — that belongs in the startup routing layer (DEV_0007).

## Scope

- **Files to create:** `src/store/authStore.ts`, `src/hooks/useAuth.ts`
- **Files to touch:** `src/services/db.ts` — add a `getUser(id)` invoke wrapper if not already present (needed for auto-login hydration)
- **Boundary:** This DEV does NOT build any UI — that is DEV_0005 and DEV_0006. It does NOT touch `App.tsx` routing — that is DEV_0007.

## Requirements / Acceptance criteria

- [ ] `authStore.ts` exports a `useAuthStore` Zustand store with shape:
  ```ts
  interface AuthStore {
    currentUser: User | null;
    isAdmin: boolean;
    login: (email: string, password: string) => Promise<void>;
    logout: () => Promise<void>;
    autoLogin: () => Promise<boolean>; // returns true if session restored, false if not
  }
  ```
- [ ] `login()` calls `login(email, password)` from `src/services/db.ts`, sets `currentUser` and `isAdmin`, and writes the returned `user.id` to `local_settings` via `saveLocalSettings` (key: `logged_in_user_id`)
- [ ] `logout()` clears `currentUser`, sets `isAdmin` to false, and removes `logged_in_user_id` from `local_settings` by saving an empty string
- [ ] `autoLogin()` reads `local_settings`, finds `logged_in_user_id`, fetches all users via `getUsers()`, and if the matching user exists sets `currentUser` and `isAdmin` — returns `true`. Returns `false` if the stored ID is empty or not found in the users list.
- [ ] `useAuth.ts` hook wraps `useAuthStore` and re-exports `currentUser`, `isAdmin`, `login`, `logout`, `autoLogin` — components never import the store directly
- [ ] `login()` throws (does not swallow) the error string returned by Rust so the UI layer can display it
- [ ] No `invoke()` calls directly in the store — all calls go through `src/services/db.ts`
- [ ] Respects `.claude/CLAUDE.md` conventions (no `unwrap()`, frontend never calls invoke() directly)

## Testing Plan

### Input
- Call `login('admin@clinic.dz', 'correctpassword')` in isolation
- Call `login('admin@clinic.dz', 'wrongpassword')`
- Call `autoLogin()` after a successful prior login (logged_in_user_id is in local_settings)
- Call `autoLogin()` with no stored ID (fresh install)
- Call `logout()` after a session is active

### Output
- Successful login: `currentUser` populated, `isAdmin` correct, `logged_in_user_id` persisted
- Failed login: promise rejects with error string, `currentUser` remains null
- `autoLogin()` with valid stored ID: returns `true`, `currentUser` populated
- `autoLogin()` with no stored ID: returns `false`, `currentUser` remains null
- Logout: `currentUser` null, `isAdmin` false, `logged_in_user_id` cleared in local_settings

### Test cases
1. Successful first login persists user ID correctly
2. Failed login does not modify state or local_settings
3. Auto-login on relaunch restores session without showing login screen
4. Logout clears all state and prevents subsequent autoLogin from succeeding
