# FEAT_0003 — Add admin-managed user onboarding and access control

## Description
The app now has login and session flow, but no way for a clinic admin to create and manage staff accounts inside the product. This feature adds user onboarding from the Settings area (admin only): create user, edit user, reset password, and set module permissions per user. This keeps access control fully local and operational for multi-user clinics without any cloud dependency.

## Scope

**In scope:**
- Admin-only Users panel in Settings for CRUD of regular users
- Admin-only password reset flow (manual reset, no email flow)
- Per-user module permissions (6 module flags) and route locking
- Schema alignment for `users` so permissions are stored per user
- Safe migration path from current schema and seeded users

**Out of scope:**
- Public self-signup screen for receptionists
- Invite-by-email or password recovery by email
- Role system beyond admin vs regular user
- Doctor lock UI redesign (kept in current model; may be separate FEAT)

## Requirements
1. Only admins can access the Users management panel.
2. Admin can create a regular user with required fields (`email`, `password`, `name`) and default denied permissions.
3. Admin can edit user profile fields and permissions without recreating the account.
4. Admin can reset a user password manually from the panel; no email reset flow exists.
5. Per-user permission flags control sidebar visibility and route access for non-admin users.
6. User records persist permissions in the `users` schema (not a single global permissions list).
7. Existing user records migrate safely with backward-compatible defaults.
8. UI strings for Users panel and permission labels are localized in `ar`, `fr`, and `en`.

## Testing Plan

### Input
- Existing database with at least one seeded admin account
- Admin logs in and opens Settings
- Admin creates two users with different permission sets
- One newly created user logs in and navigates app modules

### Output
- Users panel visible only to admin
- New users are created and persisted in `users` table
- Permission changes immediately affect module access after login/session refresh
- Password reset updates login behavior correctly
- Existing pre-migration users still log in and receive safe default permissions

### Test cases
1. **Happy path:** Admin creates receptionist user, enables Patients + Calendar only, user logs in and sees only allowed modules.
2. **Edit path:** Admin updates user name and permissions, changes are reflected on next app session.
3. **Password reset:** Admin resets password, old password fails and new password succeeds.
4. **Migration path:** Existing rows without permissions field receive default permission payload without crashing auth flow.
5. **Safety check:** Non-admin user cannot open Users panel via direct navigation.
