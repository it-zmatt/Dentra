# FEAT_0002 — App shell, navigation, and authentication

## Description

Every page in the app is currently an unreachable 6-line stub because there is no login screen, no sidebar navigation, and no session state. This feature builds the skeleton the entire product runs inside: a login screen backed by the existing `login` Rust command, a Zustand auth store with session persistence (auto-login on relaunch), a collapsible icon sidebar for the 7 main modules, and a startup routing layer that gates the app behind license validity and authentication. Once this feature is done, every future page feature can be implemented and actually reached by a user.

## Scope

**In scope:**
- Login screen (email + password, error feedback, FluentUI form, RTL layout)
- Auth Zustand store — `currentUser`, `isAdmin`, `login`, `logout`, `autoLogin`
- Session persistence — store logged-in user ID in `local_settings` so the app auto-logs in on next relaunch
- NavSidebar component — icon sidebar for 7 modules, active highlight, permission-flag aware, RTL border flip
- Startup routing in `App.tsx` — license gate check (grace period / valid key), then auth gate, then app shell
- License gate screen — shows machine ID + key entry field when the grace period has expired and no valid key is stored

**Out of scope:**
- User management UI (creating, editing, deleting users) — belongs in Settings FEAT
- Permission configuration UI — belongs in Settings FEAT
- First-launch setup wizard — separate FEAT
- Actual page implementations (patients, calendar, etc.) — each has its own FEAT

## Requirements

1. If no valid license is stored and the 7-day grace period has expired, the app shows a license gate screen with the machine ID displayed and a key entry field; all other screens are unreachable.
2. If no user is logged in (no persisted session or session is invalid), the app shows the login screen; the app shell is unreachable without logging in.
3. Entering correct credentials on the login screen calls the existing `login` Rust command, stores the returned `User` in auth store, and persists the user ID to `local_settings` so the next relaunch auto-logs in.
4. Entering incorrect credentials shows an inline error message ("Email ou mot de passe incorrect") without crashing or navigating away.
5. The app shell sidebar renders icon buttons for all 7 modules: Patients, Calendar, Doctors, Lab Work, Expenses, Statistics, Settings.
6. The active page icon is visually highlighted in the sidebar.
7. Sidebar icons for modules the current user lacks permission to access are either hidden or rendered disabled — determined by the `permissions` array in global settings.
8. The sidebar layout is RTL-aware: the sidebar appears on the right side and the main content fills the left when `locale = 'ar'`.
9. A logout button in the sidebar clears the auth store, removes the persisted user ID from `local_settings`, and returns the user to the login screen.
10. All UI text (labels, errors, placeholders) uses i18n keys — no hardcoded French or Arabic strings.

## Testing Plan

### Input
- Fresh app launch with no stored credentials
- App launch with valid stored session (auto-login path)
- Login attempt with wrong password
- Login attempt with correct credentials
- Nav between modules via sidebar
- App launch with expired license and no valid key stored

### Output
- Login screen shown on fresh launch; app shell shown after successful login
- Auto-login skips the login screen and lands directly in the patients page
- Wrong-password attempt shows inline error; credentials fields are not cleared
- Correct login stores session and shows the app shell
- Clicking a sidebar icon navigates to the corresponding page stub
- Expired-license launch shows machine ID + key entry screen before anything else

### Test cases
1. **Happy path — first login:** Fresh launch → login screen → enter valid credentials → app shell with Patients as default page → sidebar icons all visible
2. **Auto-login:** Close and relaunch app after successful login → login screen is skipped, lands directly on last active page (or patients default)
3. **Wrong password:** Enter wrong password → inline error message appears, no navigation, fields retain input
4. **No permission module:** Log in as a non-admin user whose `permissions[2]` (Doctors) is false → Doctors icon is hidden or disabled in sidebar
5. **Logout:** Click logout → auth store cleared → login screen shown → auto-login does NOT happen on next relaunch
6. **License expired:** Simulated expired grace period + no valid key → license gate screen shown → machine ID visible → entering valid key dismisses gate and proceeds to login
