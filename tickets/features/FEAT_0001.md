# FEAT_0001 — Add USB backup export with zip archive

## Description

Admin users need to export clinic data to USB drives for offsite backup and security. The backup must include both the SQLite database (`data.db`) and all appointment photos stored in the `photos/` directory. A single-file `.zip` archive named `dental_backup_YYYY-MM-DD.zip` is created and written to an admin-selected folder (typically a USB drive mounted on the Windows machine). The zip archive preserves the directory structure so data and photos remain bound together in one portable backup file.

## Scope

**In scope:**
- Export SQLite database + photos directory to a zip file
- Admin-triggered file picker to select destination folder (USB drive)
- Automatic zip filename with today's date: `dental_backup_YYYY-MM-DD.zip`
- Success/error notification to the admin
- Integration into Settings screen under a new Backups section
- Rust backend command to perform the zip operation (frontend never touches filesystem directly)

**Out of scope:**
- Restore from backup (will be handled in a separate FEAT)
- Encryption of the zip file
- Incremental backups or backup scheduling
- Cloud backup integration
- Photo deduplication or compression

## Requirements

1. Admin can click an "Export Backup" button in the Settings panel under a Backups section
2. Clicking the button opens a native folder picker dialog where the admin selects a destination directory (typically a USB drive)
3. A Rust command receives the destination folder path and creates a zip archive
4. The zip archive is named `dental_backup_YYYY-MM-DD.zip` (date in ISO format)
5. The zip contains the entire `photos/` directory tree and the `data.db` file at the archive root (no intermediate subdirectory)
6. After successful export, a confirmation message is shown with the full path to the backup file
7. If the export fails (e.g. destination is read-only, disk full), a clear error message is shown to the admin
8. The export operation is synchronous from the frontend's perspective (blocking UI with a loading overlay)

## Testing Plan

### Input
- Admin is logged in with admin role
- Clinic has at least one appointment with one or more photos, and multiple patients/doctors/expenses in the database
- USB drive is plugged in and mounted as an accessible folder on Windows
- Admin navigates to Settings → Backups panel
- Admin clicks "Export Backup"

### Output
- Native folder picker opens and shows available drives including USB
- Admin selects USB drive's backup folder and confirms
- A loading overlay shows briefly while zip is created
- Confirmation dialog appears with message: "Backup saved to [full path]/dental_backup_2026-04-26.zip"
- File browser confirms the zip file exists at that path
- Zip archive contains: `data.db`, `photos/` directory with all appointment photo subdirectories and images

### Test cases
1. **Happy path:** Admin selects USB drive, backup is created successfully, file size > 5MB (accounts for database + photos), can be unzipped on another machine
2. **Read-only destination:** Admin selects a read-only folder; error message appears: "Cannot write to selected folder. Check permissions and try again."
3. **Disk full:** Admin selects destination with <1MB free space; error message appears: "Insufficient disk space for backup. Check USB drive and try again."
4. **No photos:** Clinic has only database records with no appointment photos; backup still succeeds and zip contains only `data.db`
5. **Cancellation:** Admin opens folder picker and clicks Cancel; no backup is created, Settings panel remains unchanged
